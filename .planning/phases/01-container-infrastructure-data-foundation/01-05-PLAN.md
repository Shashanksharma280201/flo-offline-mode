**Health check function:**
```typescript
export async function healthCheck(req: Request, res: Response): Promise<void> {
  const health: any = {
    uptime: process.uptime(),
    timestamp: Date.now(),
    status: 'ok',
  };

  try {
    // Check MongoDB connection
    if (mongoose.connection.readyState !== 1) {
      health.mongodb = 'disconnected';
      health.status = 'degraded';
    } else {
      await mongoose.connection.db.admin().ping();
      health.mongodb = 'connected';
    }

    // Check Redis connection using ioredis
    const redis = new Redis({
      host: process.env.REDIS_HOST || 'localhost',
      port: parseInt(process.env.REDIS_PORT || '6379'),
    });

    const pong = await redis.ping();
    if (pong !== 'PONG') {
      health.redis = 'unhealthy';
      health.status = 'degraded';
    } else {
      health.redis = 'connected';
    }

    await redis.quit();

    if (health.status === 'degraded') {
      res.status(503).json(health);
      return;
    }

    res.status(200).json(health);
  } catch (error: any) {
    health.status = 'unhealthy';
    health.error = error.message;
    res.status(503).json(health);
  }
}
```

**Why ioredis API differences:**
- ioredis uses `new Redis({ host, port })` constructor
- ioredis ping() returns Promise<'PONG'> string, not Promise<void>
- ioredis quit() disconnects (like redis close())
- No need for separate connect() call - ioredis connects lazily on first command

**Why mongoose.connection.readyState !== 1:**
Mongoose readyState values: 0=disconnected, 1=connected, 2=connecting, 3=disconnecting. Only 1 is fully healthy.

**Why 503 for degraded/unhealthy:**
HTTP 503 Service Unavailable signals to Docker and load balancers that container is not ready to serve traffic.

**CRITICAL:** This is plain JavaScript (not TypeScript) so it can run directly via `node healthcheck.js`.

**File content:**
```javascript
// Standalone health check script for Docker HEALTHCHECK
// Exit 0 = healthy, Exit 1 = unhealthy

import http from 'http';

const options = {
  hostname: 'localhost',
  port: process.env.PORT || 3000,
  path: '/health',
  method: 'GET',
  timeout: 5000,
};

const req = http.request(options, (res) => {
  if (res.statusCode === 200) {
    process.exit(0); // Healthy
  } else {
    process.exit(1); // Unhealthy
  }
});

req.on('error', () => {
  process.exit(1); // Unhealthy
});

req.on('timeout', () => {
  req.destroy();
  process.exit(1); // Unhealthy
});

req.end();
```

**Why plain JavaScript (not TypeScript):**
- Dockerfile HEALTHCHECK needs lightweight script that runs without compilation
- TypeScript compiler only processes .ts files, not .js
- healthcheck.js can run directly: `node healthcheck.js`

**Why HTTP request to /health:**
- Verifies entire stack (Express server + MongoDB + Redis)
- Single point of failure detection
- Matches Docker health check pattern from research

**Why exit codes:**
- 0 = healthy (Docker marks container as healthy)
- 1 = unhealthy (Docker marks container as unhealthy)
- Docker uses exit codes, not stdout/stderr

**Server startup:**
```typescript
const server = app.listen(PORT, async () => {
  console.log(`Server running on port ${PORT}`);

  // Connect to MongoDB
  await connectDatabase();

  // Verify Redis configuration
  try {
    await verifyRedisConfig();
  } catch (error) {
    console.error('Redis verification failed, but continuing startup:', error);
  }

  console.log('All systems ready');
});
```

**Graceful shutdown handler:**
```typescript
// Graceful shutdown on SIGTERM
process.on('SIGTERM', async () => {
  console.log('SIGTERM received, shutting down gracefully...');

  // 1. Stop accepting new connections
  server.close(() => {
    console.log('HTTP server closed');
  });

  // 2. MongoDB connection will be closed by database.ts SIGTERM handler
  // No need to duplicate here - database.ts already has it

  // Note: database.ts SIGTERM handler will exit process after MongoDB close
});

// Also handle SIGINT for development (Ctrl+C)
process.on('SIGINT', async () => {
  console.log('SIGINT received, shutting down gracefully...');

  server.close(() => {
    console.log('HTTP server closed');
    process.exit(0);
  });
});
```

**Why this structure:**
- Research Pattern 3 shows SIGTERM handler must close server THEN database
- database.ts already has SIGTERM handler for mongoose.connection.close
- server.ts closes HTTP server, database.ts closes MongoDB (separation of concerns)

**Why await connectDatabase in listen callback:**
- Server starts listening immediately (passes Docker health check quickly)
- Database connection happens async (with retries)
- Health endpoint will report degraded until MongoDB connected

**Why catch on verifyRedisConfig:**
Redis verification is a warning check, not a startup blocker. Server should start even if Redis config verification fails (development scenario).

**Why .js extensions in imports:**
ES modules require explicit file extensions. TypeScript compiles to .js files, so imports use .js extension (not .ts).
