**Connection function with retry logic:**
```typescript
export async function connectDatabase(retryCount = 0): Promise<void> {
  try {
    await mongoose.connect(MONGO_URI, {
      maxPoolSize: 10,
      serverSelectionTimeoutMS: 5000,
      socketTimeoutMS: 45000,
    });
    console.log(`MongoDB connected: ${mongoose.connection.host}`);
  } catch (error) {
    console.error(`MongoDB connection error (attempt ${retryCount + 1}/${MAX_RETRIES}):`, error);

    if (retryCount < MAX_RETRIES) {
      console.log(`Retrying in ${RETRY_DELAY / 1000}s...`);
      await new Promise(resolve => setTimeout(resolve, RETRY_DELAY));
      return connectDatabase(retryCount + 1);
    } else {
      console.error('Max MongoDB connection retries reached. Exiting.');
      process.exit(1);
    }
  }
}
```

**Graceful shutdown handler:**
```typescript
process.on('SIGTERM', async () => {
  console.log('SIGTERM received, closing MongoDB connection...');
  await mongoose.connection.close(false);
  console.log('MongoDB connection closed through app termination');
  process.exit(0);
});
```

**Why these specific Mongoose options:**
- maxPoolSize: 10 - Reasonable for single-container deployment (research pattern)
- serverSelectionTimeoutMS: 5000 - Fail fast if MongoDB not available (allows retry logic)
- socketTimeoutMS: 45000 - Keep connections alive during long queries

**Why 5 retries with 5-second delay:**
- MongoDB takes 20-40 seconds to start (research Pitfall 5)
- 5 retries * 5 seconds = 25 seconds total retry period
- Covers typical MongoDB startup time with margin

**Why mongoose.connection.close(false):**
- false parameter: Don't force close (allow in-flight operations to complete)
- Research Pattern 3 shows this prevents data corruption during shutdown

```typescript
export async function verifyRedisConfig(): Promise<void> {
  const redis = new Redis({
    host: REDIS_HOST,
    port: REDIS_PORT,
  });

  try {
    // Verify maxmemory-policy is noeviction (BullMQ requirement)
    const policy = await redis.config('GET', 'maxmemory-policy');

    if (policy && policy[1] !== 'noeviction') {
      console.warn('WARNING: Redis maxmemory-policy should be "noeviction" for BullMQ');
      console.warn(`Current policy: ${policy[1]}`);
    } else {
      console.log('Redis maxmemory-policy verified: noeviction');
    }

    await redis.quit();
  } catch (error) {
    console.error('Redis connection verification failed:', error);
    throw error;
  }
}
```

**Why ioredis instead of redis package:**
- package.json (Plan 02 Task 1) installs ioredis ^5.10.0
- Research recommends ioredis for BullMQ (better performance, simpler API)
- ioredis uses `new Redis({ host, port })` constructor, not `createClient()`
- ioredis config() returns array: ['key', 'value'], not object

**Why this structure:**
- Research Code Example lines 619-630 shows BullMQ verifyRedisConfig pattern
- redisConnection export will be used by BullMQ Queue/Worker in Phase 4
- Verification ensures docker/redis.conf settings applied correctly

**Why warn instead of exit:**
Development might run without redis.conf mounted. Production deployment must have noeviction, but this shouldn't block startup during Phase 1 testing.
